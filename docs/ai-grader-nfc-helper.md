# AI Grader NFC identity and workstation helper

## Scope and security meaning

The AI Grader NFC feature supports two additive static-link programming profiles while preserving a future cryptographic seam:

- `NTAG215` / `ntag215_direct_pcsc_v1`: the existing direct ACR1552U PC/SC write and complete readback workflow. It does not permanently lock the tag.
- `FEIJU_F8215` / `gototags_manual_start_v1`: the GoToTags Desktop workflow that writes, verifies, permanently locks, and verifies the exact F8215 link after one operator **Start Encoding** click.
- `NTAG424_DNA` / `ntag424_dna_unimplemented`: an intentionally unimplemented future seam.

Both implemented profiles write one static NFC Forum URI record:

`https://collect.tenkings.co/nfc/{serverGeneratedPublicTagId}`

`publicTagId` is generated by the hosted service from 24 random bytes (192 bits), is exactly 32 base64url characters, and cannot be supplied by the browser or helper. The public route reconstructs the report action from the durable report relation; the tag is never an open redirect.

An NTAG215 or F8215 static URL is a convenience identity link and can be copied. Approved product language is **Registered Ten Kings NFC link**, **Write-protected registered NFC link** for a verified F8215 lock, and **Linked to this Ten Kings AI Grader report**. It is not cryptographic authentication of the chip, slab, or card. `static_url_v1` therefore reports `registered_link`, never `cryptographically_verified`.

Hosted activation additionally requires an ECDSA P-256 operational attestation from an approved finishing-workstation helper. For NTAG215 that signature proves only that the allowlisted helper reported a successful direct PC/SC write/readback (or an exact already-programmed readback) for the hosted attempt. For F8215 it proves only that the helper validated the bounded GoToTags terminal result for the exact attempt, full URL readback, and permanent write-protection state. Neither profile provides cryptographic chip, card, or slab authentication.

The schema has an intentionally unimplemented `NTAG424_DNA` / `ntag424_sun_v1` strategy seam. This release has no SUN verifier, counters, keys, originality-signature validation, or cryptographic-success state. Future key material must not be stored in application JSON.

## Hosted architecture

- `AiGraderNfcTag` owns exact tenant, published report, confirmed CardAsset, Item, certificate, and AiGraderLabel linkage; a SHA-256 UID fingerprint is stored only after successful local readback. Raw UID bytes are not accepted or persisted.
- `AiGraderNfcProgrammingAttempt` stores a token hash and a SHA-256 challenge hash, never the opaque token or raw attestation challenge, and expires within 30 minutes (the hosted route currently issues five-minute attempts). Init replay with the same actor/report idempotency key reconstructs the same one-time token/challenge; it never allocates a second live attempt.
- Completion accepts only helper protocol `tenkings-ai-grader-nfc-loopback-v2` and an ECDSA P-256/SHA-256 IEEE-P1363 signature from the current server allowlist. Legacy NTAG215 accepts only exact attestation v1 result `write_verified_pcsc_readback` or `already_programmed_exact`. F8215 accepts only exact attestation v2 result `write_locked_verified_gototags_readback` with `permanently_read_only_verified` and the originally reserved F8215 profile. The database service reconstructs and verifies the canonical statement itself; a browser flag cannot activate a registration.
- `AiGraderNfcAuditEvent` is append-only through database triggers. Revoke requires a reason; replacement revokes the old row before allocating a new public identity.
- Report-lifecycle advisory locks and partial unique indexes enforce one open reservation/active link per report, CardAsset, and Item. A separate unique index prevents the same UID fingerprint from being active on more than one card.
- Hosted endpoints are `/api/admin/ai-grader/nfc/status`, `/init`, `/complete`, `/revoke`, and `/replace`. Status/init/complete require a human `ai_grader_operator` or `ai_grader_admin`; revoke/replace require a human `ai_grader_admin`. Client role fields are ignored and service-account NFC calls fail closed.
- Ordinary init after any revoked registration fails with `AI_GRADER_NFC_REPLACEMENT_REQUIRED`. Only the admin replace transaction can revoke the exact old registration with its immutable reason/audit tuple and pass the module-private authorization that creates one replacement.
- `/nfc/[publicTagId]` is a production-database-only public read. Only a fully active, exactly linked, still-published public report resolves. Other states expose no private IDs, UID evidence, local paths, storage/provider details, or hardware controls. If the additive NFC migration is absent or persistence cannot be checked, the route returns a clean, no-store/noindex HTTP 503 unavailable page instead of claiming that a real tag is invalid.
- `/ai-grader/nfc?reportId=...` is the only browser hardware workflow. `/ai-grader/finish` shows the safe NFC state and links to it but remains usable without local hardware.

The NFC inventory policy is additive and defaults off:

```dotenv
AI_GRADER_NFC_REQUIRED=false
AI_GRADER_NFC_PROGRAMMING_ENABLED=false
AI_GRADER_NFC_ATTEMPT_TOKEN_SECRET=<random server-only value of at least 32 bytes>
AI_GRADER_NFC_WORKSTATION_PUBLIC_KEYS_JSON={<lowercase-sha256-der-spki-key-id>:{"tenantId":"ten-kings","algorithm":"ecdsa-p256-sha256-p1363","publicSpkiDerBase64":"<standard-base64-der-spki>"}}
```

`AI_GRADER_NFC_PROGRAMMING_ENABLED` defaults false. While false, authenticated status and admin revocation remain available, but init, complete, and replace return a fixed disabled error and the browser never contacts the helper. NTAG215 and F8215 are explicit profiles under this single programming gate; there is no redundant hosted or local profile-enable switch. When programming is enabled, the attempt-token secret and at least one valid P-256 key for the exact tenant are mandatory. The JSON allowlist is server-only, limited to 16 KiB/eight exact entries, and contains public SPKI only—never a private key, token, challenge, or signature.

Authenticated readiness returns only `nfcSchemaReady`, `nfcProgrammingEnabled`, `nfcRequired`, `nfcAttemptTokenConfigured`, `nfcWorkstationAttestationConfigured`, `nfcWorkstationKeyCount`, and `expectedNfcHelperProtocolVersion`. The schema probe distinguishes an unapplied NFC migration from an unexpected database failure without returning database details. NFC status reports unavailable when the schema is absent; NFC mutations return the stable schema-unavailable error, and an unexpected probe failure returns a separate stable check-failed error.

When `AI_GRADER_NFC_REQUIRED` is false, the current Finish/report/Publish/QR/inventory workflow does not depend on the NFC tables and remains operational if the migration has not been applied. When true, Add To Inventory rechecks schema readiness and one exact active non-revoked NFC row inside the existing report-locked transaction; missing or unverifiable persistence fails closed. Do not enable it before the migration is applied and the finishing helper is installed and accepted. NFC data is additive to label/report contracts; it does not remove, hide, or replace the existing QR or create another label-sheet slot.

## Operational attestation contract

Each programming attempt has an independent random 32-byte base64url challenge. Init returns the raw challenge to the authenticated operator workflow, but the database stores only its SHA-256 digest. The helper signs these exact UTF-8 lines with newline separators and no trailing newline:

```text
ai-grader-nfc-helper-attestation-v1
attemptId
attestationChallenge
publicTagId
normalizedUrl
uidFingerprintSha256
readbackPayloadSha256
readerResultCode
helperProtocolVersion
observedAt
```

The existing NTAG215 v1 statement remains byte-for-byte supported during rollout. F8215 uses the additive v2 statement:

```text
ai-grader-nfc-helper-attestation-v2
attemptId
attestationChallenge
publicTagId
normalizedUrl
chipType
securityMode
programmingProfile
adapterIdentity
adapterVersion
uidFingerprintSha256
readbackPayloadSha256
writeProtectionState
readerResultCode
helperProtocolVersion
observedAt
```

For v2 the exact profile is `FEIJU_F8215` / `static_url_v1` / `gototags_manual_start_v1`, adapter `gototags_desktop` version `4.37.0.1`, protection `permanently_read_only_verified`, and result `write_locked_verified_gototags_readback`. A v2 signature is created only after the terminal callback proves the exact requested and fully read-back URL plus static lock state. A nonterminal read event, browser boolean, or operation launch cannot be signed.

The signature algorithm is ECDSA P-256 with SHA-256 and the IEEE P1363 fixed-field 64-byte representation, encoded as unpadded base64url. `observedAt` is strict millisecond UTC RFC3339 and is checked against the attempt window with no more than two minutes of clock skew. The database service, not a route or browser boolean, checks the stored challenge digest, exact actor/tenant/report/CardAsset/Item/label/certificate/URL/digests, current helper protocol, eligible result, allowlisted tenant key, time window, and signature before consuming the attempt.

The workstation key ID is the lowercase SHA-256 digest of the DER SubjectPublicKeyInfo. The server accepts only the fixed `ecdsa-p256-sha256-p1363` algorithm and a DER SPKI that imports as the NIST P-256 curve and hashes to its allowlist key. Removing a key from the server allowlist revokes it for later completion; key bodies and IDs are not exposed by public NFC/report responses.

Completion evidence is bounded and internal: schema, key ID, algorithm, statement SHA-256, signature, observed time, protocol, and result code, plus `workstationOperationalAttestation=true` and `cryptographicTagAuthentication=false`. The raw UID, raw challenge, private key, helper token, device identifiers, paths, and APDUs are never stored in that evidence or returned publicly.

## Hardware and memory boundaries

Primary references:

- [NXP NTAG213/215/216 data sheet](https://www.nxp.com/docs/en/data-sheet/NTAG213_215_216.pdf)
- [ACS ACR1552U reference manual](https://www.acs.com.hk/download-manual/13473/REF-ACR1552U-Series-1.06.pdf)
- [ACS ACR1552U product page](https://www.acs.com.hk/en/products/575/acr1552u-usb-nfc-reader-iv/)

The helper uses Windows PC/SC (`winscard.dll`) in its standalone .NET 8 Windows process. No PC/SC/native dependency is imported into Next.js, Vercel, or the capture helper. One physical ACR1552U can expose both `PICC` and `SAM` PC/SC endpoints. The helper selects only an exact-token `ACR1552` + `PICC` endpoint, explicitly excludes `SAM`, and still fails closed unless exactly one matching PICC endpoint exists. This accepts one physical reader's contactless endpoint without mistaking its SAM endpoint for a second reader; two physical ACR1552U PICC endpoints still produce `multiple_readers`, and ACR1252 endpoints remain unsupported.

Windows can report an empty ACR1552U PICC endpoint from the **initial** `SCardConnect` as either `SCARD_E_NO_SMARTCARD` (`0x8010000C`) or `SCARD_W_REMOVED_CARD` (`0x80100069`). Only at that pre-transaction boundary, both results mean that the selected ACR1552U is connected and PC/SC-ready with `TagState=absent`; opening an operation returns the stable retryable `no_tag` error before any signature, write, or uncertain-operation state exists. This is a context-specific initial-connect classification, not a global redefinition of `SCARD_W_REMOVED_CARD`.

After a card connection or PC/SC transaction succeeds, `SCARD_W_REMOVED_CARD` continues to mean removal during the active operation and remains `tag_removed_mid_operation` or the existing protected uncertain write failure. Its operation gate cannot be released in a way that permits an overlapping retry with an unknown physical outcome. `SCARD_E_READER_UNAVAILABLE` (`0x80100017`) is never an empty-tag result; it remains `reader_disconnected` during initial connection handling. Reader selection, ACR1552U identity, native command boundaries, and all NTAG215/F8215 attestation and recovery contracts are otherwise unchanged.

For an NTAG215, pages 0-2 contain manufacturer/static-lock data and page 3 is the Capability Container. Although total user memory is 504 bytes, the standard `E1 10 3E 00` CC advertises a 496-byte NDEF data area, so this helper reads/writes NDEF only on pages 4-127. It never writes pages 0-3, user pages outside that advertised NDEF area, dynamic-lock page 130, configuration pages 131-134, OTP, passwords, or irreversible lock state.

The fixed ACR1552U native pass-through commands are implemented internally only:

- `GET_VERSION`: `FF 00 00 00 01 60`
- four-page `READ`: `FF 00 00 00 02 30 pp`
- one-page `WRITE`: `FF 00 00 00 06 A2 pp d0 d1 d2 d3`, with native `0A` ACK required
- UID read for immediate hashing: `FF CA 00 00 00`

There is no arbitrary APDU route or UI. The helper validates the exact NTAG215 `GET_VERSION` response, the CC/memory access bits, the Type 2 TLV structure, the single short NFC Forum URI record, and the exact production URL. Writes commit a zero NDEF length first, write bounded data pages, commit the final length last, and read/parse the full NDEF area back before returning success. Different nonblank content produces an observed-payload digest and requires one confirmation tied to the same attempt; changed content invalidates the confirmation.

Windows PC/SC exposes the selected card but not a reliable physical tag count. The real backend uses one reader, an exclusive card connection, and a PC/SC transaction, and reports `multipleTagDetectionSupported=false` with `pcsc_selected_card_only`. Fake-backend tests fail on multiple tags. Do not claim physical multi-tag discrimination until the separate ACR1552U hardware gate confirms behavior; operators must place exactly one tag.

### FEIJU F8215 GoToTags adapter

GoToTags Desktop `4.37.0.1` identified the qualified Shanghai Feiju sample as `F8215`, NFC Forum Type 2, with a 496-byte NDEF capacity. It is not NTAG215 and must never be stored or reported as NTAG215. The adapter uses GoToTags through its reviewed app-derived operation file and documented Variable Encode workflow; it does not add proprietary Feiju APDUs or direct F8215 PC/SC commands to the helper.

The reviewed operation permits exactly one URI record, one correlation identity, the exact Ten Kings public URL, Verify After Write, Permanent Lock, and one loopback HTTP integration. The operator keeps the tag off the reader, selects **Confirm Fresh F8215 & Prepare** in Ten Kings, clicks **Start Encoding** once in the opened GoToTags operation, and then places one fresh F8215 tag on the ACR1552U. The first click records `operator_fresh_inventory_confirmation_v1`. GoToTags performs the encode, complete readback verification, and lock. The helper accepts only one terminal `VERIFIED` callback that also matches the correlation, exact requested and read-back URL, FEIJU/F8215/Type 2/496-byte profile, ACR1552U PCSC2 reader, approved application version, and static locked state.

Initial-release fresh-tag SOP is an audited operational inventory control, not an electronic blank check:

- Take exactly one unused F8215 from the controlled unused-tag supply.
- Keep it off the reader until GoToTags requests it.
- Never return a failed, interrupted, uncertain, previously presented, written, or locked tag to unused inventory.
- Put every failed or uncertain tag in the separate quarantine container.
- Never claim that Ten Kings or GoToTags electronically proved the tag was blank.

The helper requires the exact reviewed template SHA-256 `31bfcca6cfd0e947d5368643a0aeed2ce730b9e0ad2ed9d0a503cfd5e5e05c3d`, exact signed launcher SHA-256 `d21adfdef57393b948ce4e6d8771f6daa215041fa27c777ef33de24057883774`, and Authenticode subject `CN=GoToTags, O=GoToTags, S=Washington, C=US`. The approved jpackage launcher has no Windows FileVersion resource, so pre-launch identity uses exact signed bytes. GoToTags' live callback reports the same approved application release as `v4.37.0.1`; the parser canonicalizes only exact `v4.37.0.1` or `4.37.0.1` to attested adapter version `4.37.0.1` and rejects every other value. It fails closed on executable hash, callback version, publisher, template, reader, chip, URL, correlation, verification, lock, duplicate, stale, size, method, content-type, Host, or loopback-peer mismatch. Any raw UID received in the callback exists only in bounded process memory long enough to hash it with SHA-256; the request buffer is cleared and the raw value is not logged, displayed, returned, or persisted. GoToTags itself displayed a tag identifier during qualification, so do not claim visual UID suppression inside the external application.

GoToTags requires its own signed-in account, internet/cloud access, a positive credit balance, and one encoding credit per interaction. Its HTTP/File integrations are documented Beta and can contain the full tag object. Ten Kings therefore documents GoToTags as an external cloud-dependent encoding adapter and does not claim local-only tag-data containment. The adapter must never receive a Ten Kings workstation token, browser token, signing key, or hosted credential.

GoToTags `4.37.0.1` also requires Windows Certificate Propagation to be stopped and disabled while it owns the reader. The helper and configuration script only verify that state; they never modify Windows services. `SCardSvr` must be running. Any persistent service-policy change requires separate workstation approval and rollback planning.

The supported terminal callback proves the final exact URL, post-write verification, and permanent lock, but it emits no independently attributable pre-write callback or blank-NDEF proof. The product owner accepts the audited fresh-unused-tag and quarantine SOP above for the initial release. That decision does not turn the assertion into electronic evidence and does not authorize silent overwrite of a previously presented or unexpected tag. Do not substitute UI automation, clipboard injection, SendKeys, or proprietary command probing.

## Loopback protocol

The helper binds only `http://127.0.0.1:47662` and accepts only:

- exact loopback peer and `Host: 127.0.0.1:47662`;
- exact browser `Origin: https://collect.tenkings.co`;
- browser routes `POST /pair`, `GET /status`, `POST /read`, `POST /write`, `POST /prepare`, `GET /operation-status`, `POST /operation-ack`, and same-route CORS preflight;
- one internal `POST /gototags/callback/{one-time-unguessable-id}` route that accepts no browser Origin or helper token and is reachable only through the exact bounded loopback callback contract;
- JSON bodies/responses no larger than 32 KiB;
- the dedicated `x-tenkings-nfc-token` for every route except one-time pairing.

The workstation token is separate from browser sign-in and from the camera capture-helper token. Pairing codes expire after ten minutes and have a persistent SHA-256-only consumption marker, so a helper restart does not permit replay. The browser stores the paired workstation token locally; it never sends it to the hosted API. That trust is browser-profile-local, so the installed NFC workstation shortcut always opens Google Chrome with the same dedicated `C:\TenKings\chrome-ai-grader-profile` already used by the canonical AI Grader Station launcher. It never delegates the one-time pairing URL to the Windows default browser/profile. Status exposes only helper/protocol version, safe ACR1552U model/capability, PC/SC readiness, tag state, busy state, and a fixed error code. Logs contain fixed event/result codes and redacted secret fingerprints, not tokens, raw UID, URL, device serial, paths, or hardware addresses.

All native reads and writes share one nonblocking operation gate. A timed-out native call retains the gate until it actually ends, preventing a retry from overlapping a still-running PC/SC transaction. Write idempotency keys return the original result and reject reuse with changed input.

F8215 jobs share that same operation gate, so a native PC/SC operation and a GoToTags job cannot overlap. `POST /prepare` validates the exact hosted attempt, challenge, public tag identity, URL, F8215 profile, expiration, and local idempotency context; writes one protected job directory; validates and launches one generated operation; and returns `awaiting_manual_start` without holding the browser request open. `POST /operation-status` returns only the sanitized phase and exact terminal evidence for that attempt. `POST /operation-ack` removes only an exact completed job. The acknowledgment is a paired-browser local cleanup signal; it does not independently prove hosted activation. The browser calls it only after authenticated hosted status returns `active` with the exact consumed attempt identity and the local completed URL/evidence matches. A crash or reload between hosted commit and local acknowledgment therefore preserves the recovery identity, cleans only the matching completed job, and leaves a different attempt protected.

One active job is allowed and its TTL cannot exceed the hosted attempt. A restart while a real write may still be active becomes uncertain and retains the gate. A `failed` or `uncertain` terminal job is never automatically reused or retried and deliberately keeps the gate closed. It requires the offline quarantine resolution command documented below; no automatic path may discard that state or claim encoding, readback, locking, or activation succeeded.

F8215 exposes `awaiting_manual_start` until GoToTags supplies its one terminal result, followed only by `completed`, `failed`, or `uncertain`. GoToTags does not provide independently trustworthy intermediate progress, so the helper and browser do not invent encoding, verification, or locking milestones. The large green success check appears only after the terminal callback proves exact readback and permanent lock and the hosted completion transaction returns the registration as active.

The current protocol is `tenkings-ai-grader-nfc-loopback-v2`. A native NTAG215 write request includes the exact attempt ID, public tag ID, challenge, URL, and local idempotency context. Only exact full readback can produce the signed `write_verified_pcsc_readback` or `already_programmed_exact` result. No-tag, reader, overwrite-required, partial-write, timeout, readback-mismatch, or other failure responses contain no signature. A failed idempotency result is preserved and is not automatically reused for another tag. Uncertain native work keeps its entry and operation gate to prevent overlap. F8215 terminal `failed`/`uncertain` state requires physical quarantine and the exact offline resolution procedure; do not present another tag or claim automatic retry success.

Production signing uses the fixed named current-user Microsoft Software Key Storage Provider key `TenKings.AiGrader.Nfc.WorkstationAttestation.v1`. It must be ECDSA P-256, signing-only, non-ephemeral, and `ExportPolicy=None`. First install creates or reuses that exact key; ordinary update validates its public identity and never rotates it. Protected helper JSON contains only the bounded key name and derived key ID. The private key never leaves Windows CNG or appears in JSON, logs, arguments, browser data, Vercel, or the database.

## Installation workflow (after rollout approval)

Prerequisites are the reviewed database migration and server environment, Windows .NET 8 runtime, and the already-installed ACR1552U PC/SC driver. The scripts detect readiness only; they never download or update a driver or firmware.

Run from an elevated PowerShell only after the workstation installation is approved:

```powershell
scripts\ai-grader-nfc\install-ai-grader-nfc-helper.ps1 -StartNow -CreateShortcut
scripts\ai-grader-nfc\status-ai-grader-nfc-helper.ps1
scripts\ai-grader-nfc\export-ai-grader-nfc-workstation-public-key.ps1
scripts\ai-grader-nfc\open-ai-grader-nfc-workstation.ps1
```

F8215 remains locally disabled after install. After the separately reviewed GoToTags/service prerequisites are present and while both the helper and GoToTags are closed, an approved operator can explicitly configure the separate local gate:

```powershell
scripts\ai-grader-nfc\configure-ai-grader-nfc-feiju-f8215.ps1 -Enable -GoToTagsExecutablePath '<reviewed GoToTags executable>'
```

Use the same command with `-Disable` to turn only the local adapter gate off. The command validates the fixed version, Authenticode publisher, pinned installed template, protected paths/ACLs, absence of active recovery state, stopped/disabled Certificate Propagation, and running Smart Card service. A bounded protected `abandoned-job-audit.jsonl` may remain; it is not active work. The command does not change a service, start/restart the helper, rotate a token or signing key, touch the reader, or remove recovery/audit evidence.

The install creates or reuses the named current-user non-exportable Windows CNG signing key, then creates only the dedicated `TenKingsAiGraderNfcHelper` scheduled task, `C:\TenKings\tools\ai-grader-nfc-helper`, protected `C:\TenKings\config\ai-grader-nfc`, and optional **Ten Kings AI Grader NFC** desktop shortcut. If the shared `C:\TenKings\tools` root is missing, install creates and protects it; if it already exists, install does not rewrite its ACL or any sibling tool. The dedicated helper subtree is always protected. It does not change the Dell capture-helper Startup shortcut/task or camera/light software. The export command emits only one server-allowlist entry with public DER SPKI, derived key ID, tenant, and algorithm; it cannot export private key material.

The installer is first-install only. If the protected config, dedicated install directory, or dedicated Scheduled Task already exists, it fails closed and directs the operator to the ordinary update command. Rerunning install is not an update mechanism and cannot silently recreate credentials, replace a running binary, or re-register the task. A failed first install removes only the install/config/task/shortcut artifacts that invocation created and reports any incomplete cleanup. It deliberately preserves the named non-exportable CNG key if key creation succeeded; the next reviewed install reuses and validates it instead of rotating it.

The task and shortcut do not point into the source checkout. Install copies bounded operational launchers into `C:\TenKings\tools\ai-grader-nfc-helper\maintenance`; the task uses the installed `start-ai-grader-nfc-helper.ps1`, and the shortcut uses the installed `open-ai-grader-nfc-workstation.ps1`, both with the stable install directory as their working directory. The open launcher discovers Google Chrome explicitly, fails closed if Chrome is unavailable, and supplies the fixed dedicated AI Grader profile plus `--new-window`; the shortcut command line remains free of the pairing code, workstation token, and hosted session. Hardware acceptance may therefore install from an isolated reviewed worktree without leaving the stopped task dependent on that worktree. Retain the worktree through review/acceptance; after a successful install its removal does not orphan start/open. A later ordinary update must still be launched from a fresh reviewed checkout because it builds the replacement and refreshes the stable maintenance payload.

## Ordinary helper update

Run only when no NFC programming operation is active:

```powershell
scripts\ai-grader-nfc\update-ai-grader-nfc-helper.ps1
```

Ordinary update is restricted to the exact `C:\TenKings\config\ai-grader-nfc\helper.json`, `C:\TenKings\tools\ai-grader-nfc-helper`, and `TenKingsAiGraderNfcHelper` task identity. It rejects traversal, alternate roots, and reparse points. Before reading protected config it verifies the ACL allowlist. The current user, local Administrators, and SYSTEM are the only accepted full-control principals; the config and helper trees may not inherit broader access.

The update sequence is fail-closed:

1. Validate the current executable as exactly approved helper v2 or v3, capture that exact prior version, and validate the protocol, protected ACLs, fixed paths, task identity, and existing named CNG public-key digest. A v2 install is not required to expose v3-only multi-profile fields.
2. Publish the replacement and stable operational launchers into a random contained staging directory, apply and verify its protected ACL tree, and require exactly helper v3 with both attestation-v1 and multi-profile-attestation-v2 capabilities through `--verify-build`. That maintenance mode exercises URL/NDEF and fake P-256/P1363 code only and reports `hardwareAccessed=false` and `productionKeyAccessed=false`.
3. Use the staged binary to export and re-hash only the existing public SPKI. This validates the configured key ID without creating, replacing, exporting, or rotating private key material.
4. If the helper was running, require an authenticated current-protocol, non-busy loopback status. A busy writer prevents the update before any stop or file replacement.
5. Only after all staged checks pass, stop the dedicated NFC task/process and snapshot hashes of the exact config bytes and ACL, config-directory ACL, pairing-consumption state and ACL, CNG key name/ID metadata, exported task XML identity, and shortcut presence/content.
6. Rename the prior install to a contained rollback directory, activate the staged directory, re-run build/key/ACL/state checks, and restore the prior running state. The task is not re-registered and the shortcut is not recreated; both continue to address the same stable installed launcher paths.
7. On any activation, validation, or readiness failure, stop the replacement, restore the prior directory, revalidate it against the exact captured prior v2-or-v3 contract, restore its prior running state, and return an explicit failure. Restoring v2 never requires v3-only fields. A successful update removes the rollback directory when safe; a cleanup failure is reported as `rollbackBackupRemoved=false` and leaves only old binaries under the dedicated tools root. Re-running v3-to-v3 remains supported.

Ordinary update never calls config initialization or key provisioning. It preserves the workstation token, pairing code and expiry, pairing-consumption marker, complete config bytes, key name/ID and private CNG key, config ACLs, Scheduled Task registration/principal/triggers/settings, shortcut state, and whether the helper was running. Refreshing the stable open launcher therefore delivers browser-profile corrections without recreating the shortcut or silently repairing/rotating pairing state. It does not install drivers or firmware and never references, stops, updates, or restarts the camera capture helper.

Maintenance commands:

```powershell
C:\TenKings\tools\ai-grader-nfc-helper\maintenance\stop-ai-grader-nfc-helper.ps1
C:\TenKings\tools\ai-grader-nfc-helper\maintenance\start-ai-grader-nfc-helper.ps1

C:\TenKings\tools\ai-grader-nfc-helper\maintenance\uninstall-ai-grader-nfc-helper.ps1
```

Credential maintenance requires at least one explicit `-RotateToken` or `-RotatePairingCode` switch; invoking it without a choice fails. `-RotateToken` is accepted only together with `-RotatePairingCode`, ensuring the new token and a fresh one-time pairing code are persisted in the same maintenance action so consumed browser trust cannot be stranded. Pairing-code-only rotation remains available. Rotation is never part of ordinary update. Uninstall validates the dedicated Scheduled Task and any existing **Ten Kings AI Grader NFC** desktop shortcut against their fixed installed launchers before mutation; it refuses to delete an unexpected shortcut. Uninstall preserves published binaries/config by default, and removal switches are explicit. Treat those switches as workstation-destructive maintenance and review their fixed paths before use.

Abandoned-job resolution is elevated, offline, and exact-attempt-only. First close GoToTags, stop the helper task/process/listener, remove the physical tag, and put it in the quarantine container. The command refuses `awaiting_manual_start` and `completed` state; only `failed` or `uncertain` qualifies. It writes a protected redacted audit entry containing an attempt fingerprint, prior phase/error, timestamp, `removed_and_quarantined`, and `encodingSuccessClaimed=false`, then deletes only that job's bounded operation file and removes the exact recovery state last. Restarting after successful resolution releases the operation gate. Never use it to acknowledge hosted activation or return a tag to unused inventory.

Helper builds containing the protected-leaf correction explicitly protect every generated operation, atomic state file, and quarantine audit leaf. A workstation with the earlier v3 build can instead contain one legacy job whose leaf files safely inherited the protected three-principal root ACL but fail the stricter maintenance assertion. Do not delete or edit those files manually and do not weaken the ordinary updater. After the corrected source is independently approved, run its bounded bootstrap command from the exact reviewed checkout:

```powershell
scripts\ai-grader-nfc\recover-ai-grader-nfc-f8215-stuck-job.ps1 -AttemptId '<exact attempt>' -Confirmation 'I removed and quarantined the exact NFC tag used for this F8215 attempt.'
```

That command accepts inherited ACLs only on the exact validated state, matching operation, and optional audit leaf; an inheriting directory, deny/foreign principal, reparse point, unexpected file, path mismatch, size violation, or attempt mismatch still fails closed. It requires GoToTags closed and the tag removed/quarantined, stops the dedicated task/process/listener, explicitly protects those leaves, restarts the installed helper once so a nonterminal job becomes `uncertain`, stops and protects the rewritten state again, invokes the existing failed/uncertain-only resolver, verifies exact cleanup and the protected audit, and leaves the helper stopped. It never claims write, readback, lock, or hosted activation success. Run the ordinary reviewed v3-to-v3 updater only after this recovery releases the local gate; the updated helper then prevents recurrence.

The normal workstation shortcut/open command reads but never rewrites protected config. It starts the dedicated task only when the helper is not already running and never restarts a healthy helper. Before opening the browser it waits a bounded five seconds for exactly one configured helper process, the validated dedicated Scheduled Task, and exactly one listening endpoint at `127.0.0.1:47662`. Windows may report that `HttpListener` endpoint as owned directly by the helper PID or by HTTP.sys under System PID 4. PID 4 alone is never accepted as helper identity: the exact configured helper process and task must also be present. Any other owner, endpoint, multiplicity, or timeout fails closed. This listener-only wait avoids a race between automatic pairing and .NET startup without making an HTTP, PC/SC, reader, or tag request; the authenticated browser `/status` check after hosted readiness remains the authoritative helper/protocol validation. The launcher always opens the programming route in Google Chrome using exact profile `C:\TenKings\chrome-ai-grader-profile` and a non-secret launcher marker in the scrubbed fragment. Before the first successful pairing it may also place the existing unexpired one-time code in that fragment for automatic local exchange; the operator must not discover, copy, or manually enter that code. After the persisted consumption marker matches, it opens without any pairing code. An expired unconsumed code—or a code consumed earlier by a different browser profile—fails closed and requires the explicit pairing-code-only `-RotatePairingCode -RestartHelper` maintenance action above, followed by one launch from the canonical NFC shortcut. If a separately authorized workstation-token rotation invalidated saved browser trust, that fresh shortcut code replaces it only after the helper returns exact `workstation_token_invalid`; timeouts and unavailability never clear trust or trigger pairing. Recovery never rotates a workstation token unless `-RotateToken` was separately and explicitly requested, and must never be hidden inside ordinary update.

## Deployment order

1. Review and explicitly approve the existing static-link migration plus the additive F8215 migrations `20260716225000_ai_grader_nfc_feiju_f8215_chip_type` and `20260716230000_ai_grader_nfc_feiju_f8215_gototags_two_click`. The separate enum-only transaction is required by PostgreSQL; do not combine the two new migrations.

3. Leave `AI_GRADER_NFC_PROGRAMMING_ENABLED=false` and `AI_GRADER_NFC_REQUIRED=false`; configure a random server-only `AI_GRADER_NFC_ATTEMPT_TOKEN_SECRET`.
4. Deploy the hosted code through the normal runbook and verify authenticated redacted readiness, feature-disabled behavior, legacy NTAG215 compatibility, admin revocation availability, and public invalid-state reads without contacting helper hardware.
5. Approve and update the standalone helper on the dedicated finishing PC, preserving its protected configuration, task, shortcut, token, and CNG signing key. Export/review only the public allowlist entry if required and pair the production-origin browser. Do not provision or rotate a key from ordinary update.
6. Configure and verify the exact GoToTags version/publisher/template, account/credits, Certificate Propagation policy, and F8215 adapter dependencies.
7. After the migration, secret, tenant key, helper, and adapter are ready, set `AI_GRADER_NFC_PROGRAMMING_ENABLED=true` and verify readiness for both explicit profiles.

9. Set `AI_GRADER_NFC_REQUIRED=true` only when the product requires successful NFC before the inventory transition.

## Production rollback plan

If NFC rollout must be rolled back, set `AI_GRADER_NFC_PROGRAMMING_ENABLED=false` and `AI_GRADER_NFC_REQUIRED=false`. Resolve any exact protected F8215 recovery job before changing adapter dependencies; the maintenance command refuses active or nonterminal recovery state. Roll back hosted/helper behavior without deleting or rebinding NFC records. The additive NFC tables and immutable audit history should normally remain so existing registrations and evidence are preserved. Removing tables or NFC data later requires a separate reviewed destructive migration, dry-run impact evidence, and explicit destructive-operation approval.
